## 1. Примеры Telegram Mini App {#примеры}

### Пример 1 — YCLIENTS (yclients.com)

**Что делает:** Крупнейший в России SaaS для салонов красоты, барберов, фитнеса (30 000+ бизнесов). Telegram-бот с частичной интеграцией Mini App для записи к мастеру.

**Сильные стороны:**
- Реальное время: слоты подтягиваются из CRM-расписания — двойная запись невозможна
- Цепочка напоминаний: Telegram-сообщения за 24ч и 2ч до визита (открываемость выше SMS)
- Карточки мастеров: фото, специализация, рейтинг прямо в потоке выбора
- Поддержка нескольких филиалов из одной точки входа
- Высокое доверие — клиенты знают YCLIENTS, бренд работает на репутацию

**Слабые стороны:**
- Это бот, а не настоящий Mini App: кнопки в чате, не UI-приложение — ощущение 2019 года
- Нет визуального каталога услуг: только текстовые строки, ни одной фотографии работы
- Воронка из 7+ шагов: выбор услуги → мастера → даты → времени → подтверждение → имя → телефон. На каждом шаге — отдельное сообщение. Пользователи уходят
- Портфолио мастера — минимально: имя и специальность, не галерея работ
- Нет кросс-продаж: никаких «добавьте педикюр» или «вы можете записаться на...»
- Закрыл бот и начинаешь сначала: нет сохранения состояния

**Что мы делаем лучше:**
- Заменяем линейный бот-флоу на TMA с сохранением состояния — пользователь идёт вперёд-назад, не теряет выбор
- Каждая услуга — визуальная карточка с фото, длительностью и ценой
- Карточки мастеров с галереей работ (можно листать перед записью)
- Прогресс-бар: «Шаг 2 из 3», а не бесконечный чат

### Пример 2 — Dikidi (dikidi.net)

**Что делает:** Конкурент YCLIENTS, популярен у независимых мастеров (ногтевой сервис, лэши, массаж). Мобильное приложение + веб-виджет + Telegram-бот.

**Сильные стороны:**
- Заточен под соло-мастера: поток настроен для бизнеса из 1 человека
- Память клиента: запоминает имя, телефон, предыдущие услуги — возвращающийся клиент проходит быстрее
- Список ожидания: если слот занят, можно встать в очередь — редкая и ценная функция
- Интеграция с личным каналом мастера: клиент из канала нажимает ссылку и попадает прямо в расписание
- Веб-виджет визуально чище, чем у YCLIENTS

**Слабые стороны:**
- Telegram-бот — чисто функциональный, ноль эстетики: чистый текст, никаких изображений
- Нет категорий и фильтров: 15 услуг — просто плоский список, никакой группировки (Маникюр / Педикюр / Дизайн)
- Длительность услуги — «спрятана» в названии строкой «Маникюр 60 мин — 1200 руб»: легко пропустить
- Нет Discovery-режима: Dikidi предполагает, что клиент уже знает, к кому идёт. Найти нового мастера нельзя
- Оплата непоследовательна: одни мастера включили предоплату, другие нет — клиент не знает заранее
- Нет онбординга для новых клиентов: первый и сотый визит — одинаковый интерфейс

**Что мы делаем лучше:**
- Discovery как основной режим: просмотр мастеров по категории, стилю
- Длительность — визуальный бейдж на каждой карточке услуги
- Стандарт оплаты виден ДО начала записи: «Этот мастер берёт предоплату 30%»
- Группировка услуг по вкладкам: Маникюр / Педикюр / Ногтевой дизайн / Коррекция

### Пример 3 — Самодельные боты независимых мастеров

**Что делает:** Тысячи соло-мастеров в России и СНГ создают свои Telegram-боты на BotFather + Python/Node. Типично: прайс-лист, кнопки выбора услуги, запрос на запись мастеру.

**Сильные стороны:**
- Личный контакт: бот конкретного мастера — ощущение прямой связи с ним
- Фото работ: некоторые боты присылают примеры портфолио прямо в чат
- Рассылки: мастер пушит «Есть окно в субботу 14:00» всем подписчикам
- Ноль абонентской платы: мастер платит только за хостинг (или ничего)

**Слабые стороны:**
- Нет стандарта: у каждого мастера бот работает по-своему. Клиент, записанный к 5 мастерам, имеет 5 разных опытов
- Нет реального времени: большинство ботов просто отправляют заявку, мастер вручную подтверждает — это не запись, это форма обратной связи
- Нет защиты от конфликтов: три человека написали на одно время — мастер жонглирует вручную
- Устаревание: если мастер не обновлял бота месяцами — клиент получает неактуальные цены и молчание
- Нет отмены: чтобы отменить запись, надо писать мастеру лично. Освободившийся слот никто не предложит другим

**Что мы делаем лучше:**
- Единый стандартный UX-паттерн, который мастер лишь кастомизирует
- Реальная запись с проверкой доступности слота
- Автоматическая отмена + мгновенное освобождение времени для других клиентов

### Пример 4 — Tapper / Wowner (QR-меню ресторанов в Telegram)

**Что делает:** Российские SaaS-сервисы для ресторанов (Tapper, Wowner) — QR-код на столике ведёт в Telegram Mini App с меню и кнопкой заказа. Прямая аналогия с каталогом услуг мастера.

**Сильные стороны:**
- Вкладки категорий сверху (Бургеры / Пицца / Салаты): мгновенный скан, постоянно видны — прямо применимо: Маникюр / Педикюр / Лэши / Брови
- Карточки позиций: большое фото, название, короткое описание, цена — идеальная визуальная плотность
- Плавающая кнопка корзины: всегда видна, показывает количество + сумму → аналог: «Ваша запись: 2 услуги, 120 мин, 3400 ₽ — Подтвердить»
- Эстетичный, приятный UI: пользователи показывают меню-боты друзьям
- Рейтинг на карточке заведения: 4.8 ⭐ — прямо переносится на карточку мастера

**Слабые стороны:**
- Нет временного измерения: доставку не нужно записывать на конкретный слот — в бьюти без этого нельзя
- Нет связи «услуга + исполнитель»: в ресторане неважно, кто повар. В бьюти мастер = продукт
- Спонтанная покупка vs. плановая запись: бургер заказывают импульсивно, маникюр — за 3-7 дней. Календарь должен быть главным, а не дополнительным

**Что мы делаем лучше:**
- Берём визуальный язык ресторанных ботов (фото-первые карточки, категорийные пилюли, плавающая «корзина»)
- Добавляем календарный слой и связку «мастер = продукт» — то, чего нет ни в одном ресторанном TMA

### Пример 5 — Медицинские/фитнес боты для записи (нишевая аналогия)

**Что делают:** Боты клиник (Специальность → Врач → Дата) и фитнес-расписание (Mindbody-стиль). Самая зрелая аналогия в части управления временными слотами.

**Сильные стороны (медицинские боты):**
- Воронка Специальность → Доктор → Дата/Время → идеально карпируется: Категория → Мастер → Дата/Время
- Карточка врача с квалификацией, опытом, отзывами пациентов
- Различие типов приёма: «Первичный / Повторный / Процедура» → аналог «Новый клиент / Постоянный клиент» (разная цена и длительность)
- Предварительная анкета («Есть ли аллергии?») → «Есть ли противопоказания для гель-лака?» — снижает вопросы до сеанса

**Сильные стороны (фитнес боты):**
- Режим «первый доступный слот»: «Кто свободен в эту субботу утром?» — а не поиск по услуге
- Повторяющаяся запись: «Каждый четверг» — очень актуально для постоянных клиентов на ноготках
- Теги стиля инструктора → теги стиля мастера: #гель #ногтевой_дизайн #французский #nude
- Политика отмены — показана ДО, а не после записи

**Слабые стороны:**
- Медицинские боты слишком формальны: бьюти-приложение должно быть тёплым и личным
- Месячный грид-календарь: слишком плотный, пользователи теряются — лучше показывать доступные дни крупно
- Фитнес не учитывает личную связь с тренером: в бьюти клиент лоялен конкретному мастеру

**Что мы делаем лучше:**
- Берём структуру воронки клиник и функции фитнеса (повтор, теги стиля)
- Добавляем тёплый личный тон вместо медицинского формализма
- Заменяем плотный месячный грид на горизонтальный свайп по доступным дням

## 3. Обязательные UI-элементы {#элементы}

### Навигация
- **Telegram BackButton** — нативная кнопка «Назад» Telegram (`WebApp.BackButton`). Не создавать свою
- **Telegram MainButton** — нативная большая кнопка снизу (`WebApp.MainButton`). Главный CTA каждого экрана
- **Bottom Tab Bar** — 4 вкладки: Главная / Услуги / Мои записи / Профиль
- **Заголовок экрана** — краткий, без дублирования навигации

### Компоненты
- **Скелетон-загрузка** — вместо спиннера при загрузке списков
- **Пустое состояние** — иллюстрация + текст + действие, если список пуст
- **Состояние ошибки** — понятный текст + кнопка «Повторить»
- **Карточка услуги** — фото (60% высоты карточки), название, длительность (бейдж), цена, кнопка
- **Карточка мастера** — фото, имя, рейтинг ⭐, специализация, краткое портфолио
- **Сетка слотов** — интерактивная; занятые / доступные / выбранные — визуально разные
- **Горизонтальный свайп дней** — удобнее плотного месячного грида
- **Бейджи статусов** — Подтверждено / Ожидание / Отменено / Завершено (цветовая кодировка)
- **Отзывы со звёздами** — только верифицированные; рейтинг виден на карточке мастера
- **Pull-to-refresh** — на всех списочных экранах

### Технически обязательно (Telegram API)
- `WebApp.ready()` — вызвать сразу при загрузке; до вызова Telegram показывает лоадинг
- `WebApp.expand()` — разворачивать на полный экран
- `WebApp.themeChanged` — слушать смену темы, перерисовывать
- **CSS-переменные Telegram** — использовать для ВСЕХ цветов (автоматическая поддержка светлой/тёмной темы):
  ```css
  background: var(--tg-theme-bg-color);
  color: var(--tg-theme-text-color);
  /* и т.д. */
  ```
- `HapticFeedback` — вибрация на нажатие кнопок, успешное действие, ошибку
- **Telegram Payments** — для предоплаты; никогда не делать свой платёжный экран поверх
- `initData.user` — предзаполнять форму именем/фамилией из Telegram-профиля
- `CloudStorage` — хранить предпочтения (любимые мастера, последняя категория)
- **Safe areas** — `safe-area-inset-*` для iPhone с чёлкой и Android с навигацией

### UX-стандарты
- Минимальный размер touch-области — 44×44px
- Воронка записи — максимум 4 экрана (Услуга → Мастер+Время → Сводка → Успех)
- Политика отмены — видна ДО подтверждения, не после
- Все действия — мгновенная визуальная / тактильная обратная связь
- Ни одного жёстко прописанного цвета — только переменные Telegram

### 7 главных дифференциаторов нашего TMA

1. **Фото прежде всего.** Каждая карточка мастера и каждая услуга начинается с фотографии. У конкурентов — только текст.

2. **Мастер = продукт.** Мы берём глубину ресторанных ботов (карточка, рейтинг, описание), но делаем акцент на личность мастера — потому что клиент покупает не «маникюр», а «маникюр у Кати».

3. **3 шага, не 7.** Воронка: Выбрать услугу → Выбрать мастера + время → Подтвердить. Конкуренты делают 7+ линейных шагов. Мы делаем 3 нелинейных с сохранением состояния.

4. **Discovery-режим.** Можно прийти, не зная, к кому хочешь, и найти мастера по стилю, специализации и рейтингу. Dikidi и YCLIENTS требуют знать мастера заранее.

5. **Постоянный клиент = 2 тапа.** «Вы были у Кати 12 февраля, записать снова?» — против полного прохождения воронки каждый раз.

6. **Дефицит работает на конверсию.** «Остался 1 слот на этой неделе» — честный сигнал, побуждающий к действию. Ни один конкурент это не показывает.

7. **Прозрачность до, не после.** Политика отмены, стоимость с объяснением, условия предоплаты — всё видно до подтверждения записи, а не в мелком шрифте после.

## 5. Экспертная оценка {#экспертиза}

> Три независимые перспективы: UX-дизайнер, Маркетолог, Telegram-разработчик.

---

### 👁 UX-ДИЗАЙНЕР: что работает, что ломает опыт

#### Типичные ошибки новичков

**Копирование структуры веб-сайта.**
Новички делают «сайт в Telegram» — хедер, футер, горизонтальные меню. TMA открывается как нижняя шторка, экран — 60-75% высоты телефона. Горизонтальные навигации и многоуровневые меню работают против пользователя. Весь контент — вертикальный скролл.

**Перегруз карточки услуги.**
Попытка вместить в карточку название + описание на 3 строки + длительность + цену + фото + кнопку. Результат — стена текста. Принцип: одна карточка — одно решение. Фото + название + цена + кнопка «Подробнее». Детали — на отдельном экране.

**Отсутствие визуального якоря при первом открытии.**
Лоадер → резкое появление. Или хуже — приветственный текст «Добро пожаловать». Первый экран должен немедленно показывать ценность: фотографии работ, топ-услуги, цены. Без вступлений.

**Форма записи с 6+ полями.**
Клиент уже в Telegram. Имя и телефон у системы есть. Нужно выбрать услугу, дату, время. Каждое лишнее поле — трение, которое убивает конверсию.

**Игнорирование нативных паттернов Telegram.**
Кастомные модальные окна, нестандартные шрифты, хардкод цветов вместо `--tg-theme-*`. TMA должно ощущаться как часть Telegram, а не вставка из другого мира.

**Запись без подтверждения и следующего шага.**
После «Записаться» — тишина. Обязателен финальный экран: «Мастер получила заявку. Ожидайте подтверждения в чате» + кнопка «Вернуться в чат».

---

#### Что обязательно должно быть

- **Фотографии работ — не просто желательно, обязательно.** Аудитория 18-40 принимает решение визуально. Минимум 2-3 фото на услугу, в идеале — карусель.
- **Цены открытые и честные.** Никаких «цена по запросу» и «от X». Если варьируется: «Новый клиент — 2500 ₽ / Коррекция — 1800 ₽». Клиент хочет решить сам, без переговоров.
- **Категории с горизонтальными табами** (Маникюр / Педикюр / Ресницы / Брови). Пользователь пришла за конкретным — не заставляй скроллить 20 позиций.
- **Слоты, не свободный ввод даты.** Календарь + кнопки времени. Снимает когнитивную нагрузку, исключает ошибки.
- **Статус записи в одном клике.** «Моя запись — суббота, 14:00, маникюр». Снижает поток вопросов мастеру «а когда у меня?».
- **Контакт и локация мастера.** Адрес с кнопкой в Яндекс Карты, фото мастера на главном экране. Для соло-мастера доверие = личность.

---

#### Что бесит пользователей (закрывают приложение)

- **Долгая загрузка без skeleton.** 3 секунды белого экрана — для Telegram-пользователя это вечность.
- **Потеря выбора при нажатии «назад».** Выбрала услугу, случайно вышла — всё сбросилось. Состояние сессии должно сохраняться.
- **Маленькие и нечёткие фото.** Превью 100×100 для бьюти — провал. Минимум — широкий формат в полную ширину экрана.
- **Кнопка «Оставить заявку» вместо «Записаться».** Слово «заявка» = неопределённость. «Записаться» — это результат.
- **Модальное поверх модального.** Карточка → «Записаться» → ещё попап → ещё форма. Каждый лишний экран в воронке = потеря части аудитории.
- **Нет обратной связи на действие.** Нажала кнопку — ничего. Нажала ещё раз — отправилось дважды. Каждое нажатие = мгновенный feedback.
- **Невозможность отменить запись через TMA.** Если отмена возможна — она должна быть в одно нажатие, не через чат мастеру.

---

#### UX-принципы для этого продукта

| Принцип | Суть |
|---|---|
| **«Шторка, не страница»** | TMA = продолжение диалога. Лёгкий интерфейс, не более 3 уровней навигации |
| **«Доверие через визуал»** | Фото работ = главный контент. Текст — вторичен |
| **«Одно решение за сессию»** | Клиент пришла записаться на маникюр. Апсейл — только после основного флоу |
| **«Telegram-нативность»** | MainButton, тема пользователя, стандартные шрифты — снижает когнитивную нагрузку |
| **«Мастер — личность»** | Фото + «Привет, я Лена, делаю маникюр 6 лет». Записываются к человеку, не в каталог |
| **«Минимум полей»** | Каждое поле = потенциальный выход из воронки. Оптимум: услуга + дата + время + кнопка |

---

#### Приоритет экранов: что делать в MVP

**Первая очередь (Core Flow — 4 экрана, 5 тапов до записи):**

1. **Главная / Каталог** — горизонтальные таб-категории + вертикальные карточки с фото
2. **Карточка услуги** — карусель фото, описание, цена/длительность крупно, `MainButton` «Записаться»
3. **Выбор даты и слота** — горизонтальный свайп дней + кнопки времени под ним
4. **Экран успеха** — подтверждение с деталями + кнопка «Вернуться в чат»

**Вторая очередь (после первых пользователей):**
- «Мои записи» с историей и кнопкой отмены
- Онбординг / экран «О мастере»
- Push-напоминание за 24 ч через бота
- Система отзывов (после 20+ записей)

**Не делать в MVP:**
- Система лояльности, онлайн-оплата, фильтры каталога, профиль клиента с редактированием

---

### 📈 МАРКЕТОЛОГ: что заставляет записаться

#### Психология принятия решения в бьюти

Бьюти — это не рациональная покупка, это **покупка эмоции и трансформации**. Клиент покупает не «маникюр за 1500 ₽», а «руки, которые захочется сфотографировать».

**Ключевые психологические драйверы:**
- **Доверие к мастеру как к человеку.** Соло-мастер → личность → записываются к конкретному человеку, чьи руки и вкус близки. Это нужно монетизировать через персональность.
- **Страх «облажаться».** Женщина 25-35 боится прийти к незнакомому мастеру и получить результат, который придётся исправлять. Снижение этого страха — главная задача карточки услуги.
- **Социальное сравнение.** «Другие уже записались / оценили» — мощнейший триггер в среде, где девушки из одних Telegram-чатов.
- **Импульс + удобство.** TMA открывается в два касания. Короткая красивая воронка → решение «здесь и сейчас», не «потом запишусь».

---

#### Элементы, повышающие конверсию

**Карточка услуги — витрина, не прайс-лист:**
```
[Фото результата — крупно]
Маникюр с гель-лаком — держится 3 недели без сколов
от 1 800 ₽ · 1,5 часа
★ 4.9 · 127 записей
[Кнопка: «Записаться»]
```

**Формулировки кнопок, которые конвертируют:**
- Вместо «Записаться на маникюр» → **«Хочу такой же»** (после фото)
- Вместо «Выбрать время» → **«Выбрать удобный день»**
- Вместо «Отправить заявку» → **«Подтвердить запись»**
- Финальная кнопка: **«Записана! Буду ждать вас ✓»** — с личным голосом мастера

**Флоу: максимум 3 шага:**
1. Выбор услуги (карточка с фото и ценой)
2. Выбор даты и времени
3. Подтверждение

Никаких форм с именем, фамилией, почтой. Telegram знает, кто пишет.

---

#### Социальное доказательство

Конкуренты его игнорируют — это главное окно возможностей.

**Что работает в TMA:**

- **Фото-отзывы в карточке услуги** — не страница «Отзывы» в отдельном разделе, а мини-галерея под описанием: 3-5 фото реальных работ с подписью «Работа для Алины, клиентка с 2022 года».
- **Счётчик записей:** «На эту услугу записались 243 раза» — конкретная цифра, а не рейтинг из звёздочек.
- **Цитата клиента под фото:** «Хожу только к Кате уже 2 года. Руки всегда на уровне — Марина». Имя + конкретика = доверие без ощущения рекламы.
- **Лицо мастера в шапке:** «Привет, я Катя. 6 лет практики, 400+ постоянных клиенток». Лицо убирает анонимность и снижает барьер первой записи.

**Что НЕ работает:** скриншоты переписок (выглядят как фотошоп), безликие «★★★★★ Отличный мастер!» без деталей.

---

#### Ценообразование: как показывать цены, чтобы они продавали

**Порядок элементов карточки (цена — после ценности):**
1. Фото результата
2. Что входит в услугу
3. Время процедуры
4. Цена (уже оправданная ценностью)

**Формулировки, снижающие ценовой барьер:**
- «от 1 800 ₽» — снижает психологический порог
- «1 800 ₽ — включает дизайн, снятие и уход за кутикулой» — дробление повышает воспринимаемую ценность
- Для премиума: «держится 3-4 недели — это 60 ₽ в день за идеальные руки»

**Якорение.** Три варианта услуги: базовый / стандарт / премиум. Средний выбирают чаще — это не манипуляция, это удобная навигация.

**Никогда:** «уточняйте цену в личных сообщениях» — мгновенно убивает конверсию.

---

#### Триггеры срочности и дефицита (без манипуляций)

У соло-мастера дефицит всегда настоящий — она физически ограничена в количестве клиентов.

- «На этой неделе осталось 2 свободных слота» — если это правда
- «Ближайшая запись — 14 марта» — конкретная дата создаёт срочность без давления
- «Популярная услуга — обычно записываются за 5-7 дней» — соцдоказательство + мягкая срочность
- Занятые слоты в календаре показывать серым, не скрывать: клиент сам видит, что мест мало

**Не делать:** таймеры «Цена действует ещё 00:14:32» — аудитория 18-40 этот трюк знает, он разрушает доверие к мастеру.

---

#### Удержание и возврат (LTV важнее первой записи)

Маникюр раз в 3 недели, брови раз в месяц. LTV клиентки за год = 12-20 визитов.

**Механики удержания:**

- **Автонапоминание:** через 3 недели бот пишет: «Прошло 3 недели — самое время обновить маникюр. Ваш любимый слот в пятницу 18:00 — свободен. Записать?» → одна кнопка.
- **Персональная история:** «В прошлый раз вы делали нюдовый дизайн — посмотрите, что новенького появилось» + фото.
- **Лояльность без карточек:** «Вы были у нас 5 раз — следующая процедура -10%». Счётчик в боте.
- **Реактивация:** если клиентка не записывалась 60+ дней — бот: «Давно не виделись! Приходите в этот месяц и получите [бонус]».

---

#### Ошибки, убивающие конверсию

| Ошибка | Почему критично |
|---|---|
| Длинная воронка (7+ шагов) | Каждый шаг = потеря 20-30% пользователей |
| Нет фото или плохие фото | В бьюти фото = 80% решения о записи |
| Прайс-лист вместо каталога | Работает как меню в столовой, а не как витрина |
| Кнопка «Узнать подробнее» | Это кнопка прокрастинации — клиент уходит думать |
| Тишина после записи | Сразу после — автосообщение от мастера с деталями |
| Обезличенность | Бот без лица мастера = запись в безымянный сервис |
| 20 услуг сразу | Начни с 3-5 популярных + «Все услуги» |

### ⚙️ TELEGRAM-РАЗРАБОТЧИК: ограничения и возможности платформы

#### Ограничения платформы

**CloudStorage:**
- Лимит: **1024 ключа** на пользователя на бота; каждое значение — **до 4096 символов** ⚠️ *Проверить актуальные лимиты на [core.telegram.org/bots/webapps](https://core.telegram.org/bots/webapps) перед разработкой*
- Это НЕ база данных — это сессионное хранилище
- Недоступен в Desktop-клиенте на старых версиях → всегда нужен localStorage как fallback:
```javascript
const storage = {
  set: (key, value) => {
    try { Telegram.WebApp.CloudStorage.setItem(key, value, () => {}); }
    catch { localStorage.setItem(`tg_${key}`, value); }
  }
};
```

**Файлы и изображения:**
- `<input type="file">` работает на iOS и Android с оговорками
- На Android WebView — есть проблемы в некоторых версиях клиента
- Для фото мастера — CDN + lazy loading, не Base64 в CloudStorage

**Telegram Payments в РФ:** ⚠️ *Платёжный ландшафт меняется — проверить актуальный список провайдеров на [core.telegram.org/bots/payments](https://core.telegram.org/bots/payments)*
- Stripe — недоступен в РФ (санкции)
- **ЮКасса** — была доступна как Telegram Payment Provider, актуальность проверить ✅⚠️
- PayMaster, Сбербанк — требовали отдельного согласования с Telegram
- **Главный риск проекта** — согласуй платёжного провайдера до начала разработки, получи `provider_token`, протестируй в тестовом режиме. Это занимает 1-2 недели

**Уведомления:**
- TMA не может самостоятельно слать уведомления — только через бота
- Нужен `requestWriteAccess()` для отправки уведомлений вне сессии:
```javascript
Telegram.WebApp.requestWriteAccess((granted) => {
  if (granted) {
    // Сохрани разрешение на сервере — бот теперь может писать пользователю
    fetch('/api/grant-notifications', { method: 'POST', ... });
  }
});
```

#### Что работает хорошо — нативные преимущества

**HapticFeedback — без разрешений, без ограничений:**
```javascript
const haptic = Telegram.WebApp.HapticFeedback;
haptic.selectionChanged();          // При выборе услуги
haptic.impactOccurred('medium');    // При «добавлении в корзину»
haptic.notificationOccurred('success'); // При успешной записи
haptic.notificationOccurred('error');   // При ошибке
```

**MainButton — нативная кнопка, сам решает проблемы safe area:**
```javascript
const btn = Telegram.WebApp.MainButton;
btn.setText('Записаться (2500 ₽)');
btn.color = '#FF6B9D';
btn.show();
btn.showProgress(true); // Во время запроса
btn.onClick(() => submitBooking().finally(() => btn.hideProgress()));
```

**initData.user — предзаполнение без форм:**
```javascript
const user = Telegram.WebApp.initDataUnsafe.user;
// { id, first_name, last_name, username, language_code }
// Телефон — через отдельный requestContact()
Telegram.WebApp.requestContact((sent, contact) => {
  if (sent) bookingForm.phone = contact.contact.phone_number;
});
```

**Нативные попапы вместо `window.confirm()`:**
```javascript
Telegram.WebApp.showPopup({
  title: 'Отменить запись?',
  message: 'Возврат предоплаты в течение 24 часов',
  buttons: [
    { id: 'cancel', type: 'destructive', text: 'Отменить запись' },
    { id: 'keep', type: 'cancel', text: 'Оставить' }
  ]
}, (buttonId) => { if (buttonId === 'cancel') cancelBooking(); });
```

---

#### Что лучше не делать (анти-паттерны)

**Не использовать `window.history` напрямую** — сломает BackButton. Нужен собственный стек роутинга.

**Не делать тяжёлые анимации** — WKWebView и WebView ограничены по GPU. Только `transform` и `opacity`:
```css
/* ПЛОХО */
.card { transition: box-shadow 0.3s, margin 0.3s; }
/* ХОРОШО */
.card { transition: transform 0.2s; will-change: transform; }
.card:active { transform: scale(0.97); }
```

**Не хранить всё в CloudStorage.** Правильное разделение:
- CloudStorage: предпочтения пользователя, текущая корзина
- localStorage: кэш каталога, изображений, расписания
- Сервер: записи, платежи, данные пользователей

**Не игнорировать изменение viewport** при открытии клавиатуры:
```javascript
Telegram.WebApp.onEvent('viewportChanged', ({ isStateStable }) => {
  if (isStateStable) {
    document.documentElement.style.setProperty(
      '--viewport-height', `${Telegram.WebApp.viewportHeight}px`
    );
  }
});
```

**Не использовать `window.alert()` / `window.confirm()`** — на iOS TMA они заблокированы или выглядят как системные.

#### Архитектурные решения для проекта

**Рекомендуемый стек:**
```
Frontend:  Vanilla JS + Vite  (меньше bundle, не нужен SSR)
Стили:     CSS Variables + CSS Modules
Backend:   Node.js + Fastify  или  Python + FastAPI
БД:        PostgreSQL + Redis (кэш расписания)
Хостинг:   Railway / Render (HTTPS обязателен)
Bot SDK:   grammY (Node) или aiogram (Python)
```

**Почему не Next.js:** SSR не нужен для TMA — контент всегда за авторизацией. Next.js добавляет значительно больше клиентского JS, чем Vite-based SPA ⚠️ *Реальный overhead зависит от конфигурации, это не абсолютный запрет — просто аргумент в пользу более лёгкого стека для MVP*.

**CSS-переменные — правильная структура:**
```css
:root {
  --tg-bg: var(--tg-theme-bg-color, #ffffff);
  --tg-text: var(--tg-theme-text-color, #000000);
  --tg-hint: var(--tg-theme-hint-color, #999999);
  --tg-secondary-bg: var(--tg-theme-secondary-bg-color, #f1f1f1);
  --safe-bottom: env(safe-area-inset-bottom, 0px);
  /* Единственный брендовый цвет: */
  --brand: #FF6B9D;
}
.page {
  /* Учитываем MainButton (60px) и safe area */
  padding-bottom: calc(60px + var(--safe-bottom));
}
```

#### Безопасность: обязательно

**Валидация initData на сервере (всегда):**
```javascript
function validateTelegramInitData(initData, botToken) {
  const params = new URLSearchParams(initData);
  const hash = params.get('hash');
  params.delete('hash');

  const dataCheckString = Array.from(params.entries())
    .sort(([a], [b]) => a.localeCompare(b))
    .map(([k, v]) => `${k}=${v}`).join('\n');

  const secretKey = crypto.createHmac('sha256', 'WebAppData').update(botToken).digest();
  const expectedHash = crypto.createHmac('sha256', secretKey).update(dataCheckString).digest('hex');

  if (hash !== expectedHash) throw new Error('Invalid signature');
  if (Date.now() / 1000 - parseInt(params.get('auth_date')) > 3600) throw new Error('Expired');
}
```

| Функция | iOS (WKWebView) | Android (WebView) |
|---|---|---|
